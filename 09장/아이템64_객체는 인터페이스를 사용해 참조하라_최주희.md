# 아이템64.객체는 인터페이스를 사용해 참조하라

적합한 인터페이스만 있다면 매개변수[아이템 51] 뿐 아니라 반환값, 변수, 필드를 전부 클래스가 아닌 인터페이스 타입으로 선언하라. (객체의 실제 클래스를 사용할 상황은 오직 생성자로 생성할 때뿐)

```java
//좋은 예 : 인터페이스를 타입으로 사용
Set<Son> sonSet = new LinkedHashSet<>();

//나쁜 예 : 클래스를 타입으로 사용
LinkedHashSet<Son> sonSet = new LinkedHashSet<>();
```

<br>

## 인터페이스를 타입으로 사용할 때 유연성 증가

프로그램이 훨씬 유연해진다.

나중에 구현 클래스를 교체하고 싶다면 그저 새 클래스의 생성자 or 다른 정적 팩터리를 호출해주기만 하면 된다.

다른 코드는 전혀 손대지 않아도 된다. 

```java
//Set<Son> sonSet = new LinkedHashSet<>();
//위 코드를 생성자만 바꾸어주고 다른 코드는 하나도 고칠 것이 없이 사용가능
Set<Son> sonSet = new HashSet<>();
```

상위 인터페이스가 있음에도 구현 클래스 타입으로 선언한다면, 해당 타입을 사용하고 있는 클래스 내부에는 구현 클래스에 대한 강한 의존관계가 생긴다. 그러므로 상위 인터페이스 타입을 사용하자.

이 때 주의할점은, 원래의 클래스가 인터페이스의 일반 규약 외의 특별한 기능을 제공하고 그 기능에 기대어 동작했다면 새로운 클래스도 같은 기능을 제공해야한다.

<br>

## 적합한 인터페이스가 없다면 당연히 클래스로 참조

1) String, BigInteger 같은 값 클래스 

- 값 클래스를 여러가지로 구현될 수 있다고 생각하고 설계하지 않기때문
- 이런 값 클래스는 매개변수, 변수, 필드, 반환 타입으로 사용해도 무방하다.

2) 클래스 기반으로 작성된 프레임워크가 제공하는 객체들

- ex) java.io 패키지의 여러 클래스 (OutputStream 등)
- 특정 구현 클래스보다는 (보통 추상 클래스인) 기반 클래스를 사용해 참조하는게 좋음

3) 인터페이스에는 없는 특별한 메서드를 제공하는 클래스들

- ex) PriorityQueue 클래스는 Queue 인터페이스에는 없는 comparator 메서드를 제공
- 이렇게 특별한 메서드를 꼭 사용해야하는 경우에만 클래스 타입을 직접 사용해야한다.

<br>

## 결론

주어진 객체를 표현할 적절한 인터페이스를 찾아서 그 인터페이스를 참조하여 더 유연하고 세련되게 하라.

적합한 인터페이스가 없다면 클래스의 계층구조 중 필요한 기능을 만족하는 가장 덜 구체적인 상위 클래스를

타입으로 사용하자.
